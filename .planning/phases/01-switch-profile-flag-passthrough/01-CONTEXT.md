# Phase 1: switch-profile flag passthrough - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

`switch-profile` gains the ability to forward arbitrary Home Manager activation flags (`-b`, `-n`, `--show-trace`, etc.) to the underlying `$RESULT_LINK/activate` script. Currently the script accepts exactly one argument (the profile name) and passes nothing to `activate`. This phase adds flag parsing so that all arguments except the profile name are forwarded verbatim.

</domain>

<decisions>
## Implementation Decisions

### Argument parsing strategy
- **D-01:** Profile name is always the **last** positional argument. All preceding arguments are forwarded to `activate`. This matches the success criteria examples (`switch-profile -b backup work`, `switch-profile --show-trace work`) and follows conventional CLI patterns (command + flags + target).
- **D-02:** No `--` separator is required. The script treats the final argument as the profile name unconditionally. Users who want to pass a flag as the last argument to `activate` can use the form `switch-profile -b backup work` where `work` is still last.

### Flag handling scope
- **D-03:** **Blind passthrough** — the script does NOT validate, parse, or enumerate known flags. All arguments except the last are forwarded verbatim to `activate`. This keeps the script forward-compatible with future Home Manager changes and avoids a maintenance burden of tracking `activate`'s flag surface.
- **D-04:** No `--help` flag. None of the existing scripts (`setup.sh`, `build-profiles.sh`, `switch-profile.sh`) implement `--help`, and adding it here would be inconsistent. The usage line printed on error is sufficient.

### Usage message
- **D-05:** Update the usage line to `Usage: switch-profile [FLAGS...] <profile-name>` — minimal, consistent with the script's existing brevity. No examples in the usage output; the README covers that.

### Error handling
- **D-06:** The existing fail-fast check ("no pre-built result for profile") remains. Profile name extraction (last arg) happens before the existence check. If no arguments are provided, print the updated usage line and exit 1 (same behavior as today).
- **D-07:** If `activate` fails due to bad flags, its exit code propagates naturally through `exec`. No wrapper error handling needed.

### Claude's Discretion
- Exact shell mechanics for extracting the last arg and collecting remaining args (e.g., array slicing vs parameter manipulation). Any idiomatic bash approach is fine as long as it handles zero-flag and multi-flag cases.
- Whether to preserve `exec` (current behavior) or use a plain invocation. `exec` is preferred since there's no cleanup needed after activation.

</decisions>

## Existing Code Insights

### Reusable Assets
- `switch-profile.sh` structure: the `DATA_DIR` / `RESULT_LINK` pattern is already correct and needs no changes — only the argument handling and `activate` invocation line change.

### Established Patterns
- All scripts use `set -euo pipefail` and `#!/usr/bin/env bash` — new code must follow this.
- Error messages follow `"switch-profile: <message>" >&2` pattern (see line 6, 14-15 of current script).
- `exec` is used for the final `activate` call — replaces the shell process. This should be preserved.
- Scripts use `${VAR:-default}` for optional env vars (see `CABANASHMUL_DIR`, `XDG_DATA_HOME` patterns).

### Integration Points
- The `activate` script inside the Nix result link is the downstream consumer of forwarded flags. It is generated by Home Manager and accepts flags like `-b`, `-n`, `--show-trace`.
- `public/packages.nix` wraps `switch-profile.sh` via `writeShellApplication` with `runtimeInputs` — no changes needed there since the script interface (stdin/stdout/exit codes) doesn't change.
